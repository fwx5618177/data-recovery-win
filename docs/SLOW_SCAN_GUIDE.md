# 扫描很慢？这份诊断指南先看

> 适用：扫描速度持续 < 1 MB/s（特别是 USB U 盘 / SD 卡 / 外置 USB 移动硬盘）。

## 短答：是硬件瓶颈，**先 dump 镜像再扫镜像**

业界（PhotoRec / FTK Imager / R-Studio / DMDE）的统一做法都是：

```
   原盘 (慢 / 可能在掉块)
       ↓ ddrescue / dd / 本工具内置的整盘镜像 dump
   镜像 .img 文件 (放在内部 SSD)
       ↓ 用本工具扫镜像（速度快一个数量级）
   恢复的文件
```

**为什么这么做？**

1. **原盘只读一次**。盘已经在掉块状态了，反复随机访问只会加速死亡。
2. **dump 是顺序大块读**，最大化盘的剩余健康带宽（避免数据恢复算法不可避免的小块随机读）。
3. **扫镜像是内部 SSD 读**，速度 = 你的 SSD 顺序读 (500 MB/s ~ 7 GB/s)，比 USB 桥接 U 盘快 100×。
4. **可重复**：扫错了 / 想换参数重扫 → 直接重扫镜像，原盘不再受罪。

## 本工具内置 dump 工具的使用

1. 主界面 → 上方 **工具** 菜单 → **整盘镜像 dump**
2. 选源盘（要 dump 的那块）+ 目标 .img 文件位置（**必须是另一块盘的内部 SSD**）
3. 等 dump 完成（速度跟盘的顺序读能力一致，128GB U盘 ~30 min 到几小时）
4. 主界面 → 选源盘那一栏 → 下方"**选择镜像文件…**"按钮 → 选刚 dump 的 .img
5. 像扫真盘一样扫镜像

## 为什么 USB 慢盘特别坑

USB 桥接 U 盘 / 移动硬盘有三层瓶颈：

1. **USB 桥接控制器**：很多桥接芯片对随机读 / 小块读支持差。理论 USB 3.0 = 5 Gbps = 625 MB/s，但廉价控制器实际只跑 5-20 MB/s。
2. **闪存控制器内部 GC / 磨损均衡**：U 盘控制器在持续读时会做后台垃圾回收，触发"读卡"现象 —— 表现为速度从 100 MB/s 突然掉到几百 KB/s。
3. **坏块 / 弱块重试**：闪存失效是渐变的，弱块读取成功但耗时 5-30 秒。OS 不报错只是慢。

数据恢复算法**必然**包含小块随机读（解析文件头 / 元数据时），所以放大了上述瓶颈。

## 用户问的常见问题

### Q1：dump 也很慢，咋办？
A：dump 速度 = 盘**顺序读**速度。如果连 dump 都慢，说明盘的物理状态已经很糟糕了。建议：

1. 接 USB 3.0 接口（不是 2.0）+ 不经过 Hub 直插主机
2. 换条好 USB 线（劣质线限速）
3. 如果可以，把 U 盘 / SD 卡的存储芯片直接焊到测试板（chip-off recovery）—— 这是数据恢复实验室级操作

### Q2：扫 .img 文件用什么工具？
A：本工具就行。**选源盘**界面下面有"选择镜像文件…"按钮，选 .img / .dd / .raw 文件即可。

### Q3：dump 时盘断开了会怎样？
A：本工具的 dump 是 ddrescue 风格 —— 自动跳过坏块、记录坏块清单。中断后可恢复继续。但**优先建议**：dump 期间不要碰盘，让它一次性跑完。

### Q4：能不能后台跑扫描，我先用电脑做别的？
A：可以。扫描跑在独立 goroutine 不阻塞 UI。但**别同时让另一个程序读这块盘** —— 会争抢有限的 USB 带宽。

## 不要做的事

- ❌ **不要在 dying 的盘上反复试**（每次扫描都让盘更接近物理死亡）
- ❌ **不要把 .img 存到同一块盘**（IO 自我吞噬）
- ❌ **不要不带 ddrescue 风格的 dump 工具去 cat / dd 全盘**（一个坏扇区就整盘卡死）
- ❌ **不要相信"修盘软件"宣传**（chkdsk / DiskGenius 的"修复"是 metadata 修复，物理坏块解决不了）

## 反馈渠道

如果上述都做了速度仍异常，UI → "导出诊断包" → 把 zip 发给我们。包里有：
- 本次扫描的 ResilientReader bad sectors 清单
- IO goroutine 平均 ReadAt 耗时
- 各阶段速度时间线

我们可以从日志判断瓶颈具体在哪一层。
